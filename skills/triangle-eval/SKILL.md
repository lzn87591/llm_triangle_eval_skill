# 三角评估系统 (Triangle Evaluation System)

## 系统概述

本Skill实现了一个三角色评估架构：
- **角色A (目标Agent(如Qoder/Claude Code/OpenClaw))**：回答者，即我自己
- **角色B (领导者)**：领导者，质疑和评估
- **角色C (验证者)**：独立审计员，客观验证

## 渐进式实现策略（推荐）

基于成本效益考量，采用三阶段渐进式部署：

### Phase 1: Prompt-based 基础版（当前）
- **适用场景**：验证核心逻辑、原型测试
- **成本**：单轮评估约 3x token（B+C两轮调用）
- **延迟**：串行约 6-10秒
- **触发条件**：所有评估请求

### Phase 2: 智能门控版（待实现）
- **适用场景**：生产环境
- **成本优化**：仅对高价值回答启用三角评估
- **触发条件**：
  - 回答长度 > 500 tokens
  - 涉及安全敏感操作
  - 用户明确要求深度评估
  - 历史评估准确率 < 80% 的领域

### Phase 3: 完整版（待实现）
- **适用场景**：高精度要求场景
- **特性**：并行执行、LangGraph状态管理、历史准确率加权
- **成本**：最高，仅对关键任务启用

## 触发条件

### 自动触发（Phase 1）
当用户说以下话语时自动触发：
- "评估我的回答"
- "请评审"
- "三角评估"
- "对这个回答进行质量检查"
- "你觉得刚才的回答怎么样"

### 智能门控（Phase 2+）
系统根据以下规则自动决定是否启用三角评估：

| 条件 | 权重 | 说明 |
|------|------|------|
| 回答长度 > 500 tokens | +30% | 长回答更可能有问题 |
| 涉及代码/架构设计 | +25% | 技术方案需要严格审查 |
| 涉及安全/权限 | +40% | 高风险场景必须评估 |
| 历史准确率 < 80% | +20% | 薄弱领域加强审查 |
| 用户明确要求 | +50% | 用户意图优先 |
| **阈值** | **≥60%** | 超过则启用三角评估 |

**低于阈值的处理**：使用轻量级自评估（单Agent多轮反思）

## 评估流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   采集上下文  │────►│  调用Agent B │────►│  调用Agent C │
│  (问题+回答) │     │  (提出质疑)  │     │  (独立审计)  │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  输出评估报告 │◄────│  B综合评分   │◄────│  C回应B的问题│
│  (给用户)    │     │  (最终裁决)  │     │             │
└─────────────┘     └─────────────┘     └─────────────┘
```

## 执行步骤

### Step 1: 采集上下文

从当前对话中提取：
- **原始问题**：用户最初提出的问题
- **回答内容**：目标Agent(如Qoder/Claude Code/OpenClaw)给出的完整回答
- **回答思路**：如果有展示思考过程，一并收集

如果上下文不完整，询问用户：
> "请确认要评估的问题是：'{question}'，我的回答是否从 '{answer_start}...' 开始？"

### Step 2: 调用 Agent B (领导者)

使用 `leader-interviewer` Agent 进行第一轮评估：

**输入给B的格式**：
```
【评估任务】
原始问题：{question}

【角色A的回答】
{answer}

【要求】
请按照领导者角色的要求，对以上回答进行质疑分析，列出3-5个质疑点，
并提出需要验证者确认的问题。
```

**B的输出**：质疑分析 + 需要C验证的问题列表

### Step 3: 调用 Agent C (验证者)

使用 `validator-auditor` Agent 进行独立审计：

**输入给C的格式**：
```
【审计任务】
原始问题：{question}

【角色A的回答】
{answer}

【角色B的质疑】
{criticisms}

【B需要确认的问题】
{questions_from_b}

【要求】
请按照验证者角色的要求，对A的回答进行独立审计，回应B的问题，
并给出你的专业判断。
```

**C的输出**：独立审计报告

### Step 4: B 进行最终评分

将C的审计报告反馈给B，让B给出最终评分：

**输入给B的格式**：
```
【最终评分任务】
原始问题：{question}
角色A的回答：{answer}

【之前你提出的质疑】
{your_criticisms}

【验证者C的审计报告】
{audit_report}

【要求】
请基于C的审计报告，给出最终评分和改进建议。
按照评分维度和输出格式要求完成。
```

**B的输出**：综合评估报告（含评分）

### Step 5: 整合输出

将B和C的输出整合成最终的评估报告呈现给用户。

**可解释性要求**：
- 每个评分必须有具体理由
- 展示B和C的置信度
- 如有分歧，明确标注并说明裁决依据
- 提供可追溯的评估路径


## 冲突解决机制（B与C分歧处理）

当B和C的评估结论存在分歧时，按以下流程处理：

### Step 1: 提取置信度
从B和C的输出中提取：
- B的置信度：confidence_b
- C的置信度：confidence_c

### Step 2: 三级仲裁

**Level 1: 置信度差值裁决**
```
if |confidence_b - confidence_c| > 0.3:
    采纳置信度高的一方
else:
    进入Level 2
```

**Level 2: 事实核查裁决**
- 分歧点涉及客观事实 → 以可验证的技术文档为准
- 分歧点涉及主观判断 → 进入Level 3

**Level 3: 标记争议**
- 在报告中明确标注："B与C在此点存在分歧"
- 展示双方观点
- 建议用户人工判断

### 分歧输出格式

```markdown
## ⚠️ 评估者分歧点

| 分歧主题 | B的观点 | C的观点 | 置信度(B/C) | 裁决结果 |
|---------|---------|---------|------------|---------|
| [主题] | [B的判断] | [C的判断] | 0.X / 0.Y | [采纳B/采纳C/存在争议] |

### 争议说明
[详细说明分歧内容和裁决依据]
```

## 成本监控与降级策略

### Token消耗估算

| 阶段 | 输入Token | 输出Token | 小计 |
|------|----------|----------|------|
| B质疑阶段 | ~1000 | ~800 | ~1800 |
| C审计阶段 | ~2000 | ~1500 | ~3500 |
| B评分阶段 | ~3000 | ~1000 | ~4000 |
| **总计** | - | - | **~9300 tokens** |

*基于GPT-4定价：约 $0.28/次评估*

### 降级策略

当评估成本过高或超时时，自动降级：

**Level 1降级（超时>5秒）**
- 跳过C的详细审计
- B直接给出快速评分
- 标记为"快速评估"

**Level 2降级（Token预算不足）**
- 仅执行B的质疑阶段
- 输出简化版评估报告
- 标记为"轻量评估"

**Level 3降级（服务不可用）**
- 直接放行，标记为"待评估"
- 记录到待处理队列
- 稍后异步完成评估

## 边界情况处理

### 上下文不足
如果无法确定要评估的内容，询问用户：
> "请提供要评估的具体问题和回答内容，或者指出这是针对我们刚才对话中的哪个回答。"

### 评估结果分歧大
如果B和C的判断严重不一致：
1. 执行上述冲突解决机制
2. 在报告中特别标注分歧点
3. 建议用户关注该分歧领域

### 多轮对话
如果评估涉及多轮问答：
1. 明确标注评估的是哪一轮
2. 如有必要，可以分别评估每轮
3. 或综合评估整体对话质量

### 评估失败处理
如果Agent调用失败：
1. 重试1次
2. 仍失败则降级到轻量评估
3. 记录失败原因到日志

## 使用示例

**用户**: "请评估你刚才关于微服务架构的回答"

**系统执行**:
1. 采集：提取用户问题和我的微服务回答
2. B质疑：找出3个潜在问题（如"服务拆分粒度"、"数据一致性"、"运维复杂度"）
3. C审计：独立验证这些质疑，发现还遗漏了"监控方案"
4. B评分：基于C的反馈，给出75分及改进建议
5. 输出：整合报告展示给用户

## 注意事项

1. **保持客观**：评估的是回答质量，不是目标Agent(如/Claude Code/OpenClaw)本身
2. **建设性**：所有批评都应有改进建议
3. **可执行**：建议要具体，不能空泛
4. **及时性**：评估应在回答后尽快进行，确保上下文新鲜
